Vector-ready scraping pipeline

Short answer: A vector-ready scraping pipeline is one whose output can be embedded into a vector database with no intermediate cleaning step. The scraper returns clean markdown rather than raw HTML, preserves a stable source URL per chunk, and emits a structure that chunkers and document loaders can ingest directly.

What "vector-ready" means in practice

Embedding models work best on coherent prose with little markup noise. Raw HTML, with its navigation chrome, script tags, and styling attributes, blows out the context window and pollutes similarity scores. A vector-ready output strips those, keeps headings as section anchors, preserves code blocks and lists, and links every chunk back to its source URL so retrieval can cite where an answer came from. The four properties that matter: clean markdown, deterministic ordering, stable URLs, and per-chunk metadata (title, h1, lastmod).

The URL → chunk → embedding flow

A working RAG pipeline looks like this:

  1. Discover — map a domain to the set of URLs you care about (sitemap parse or BFS crawl).
  2. Scrape — fetch each URL and return clean markdown.
  3. Chunk — split markdown by heading or by token window (e.g. 800 tokens with 100 overlap).
  4. Embed — call your embedding model per chunk.
  5. Upsert — write {id, vector, source_url, title, text} to your vector store.

The scraper is the bottleneck for quality. If it returns HTML, you spend engineering time writing readability heuristics; if it returns clean markdown, the rest of the pipeline is glue code.

Frequently asked

Why markdown and not plain text?
Markdown preserves heading structure, code blocks, and lists, which are strong signals for chunk boundaries. Plain text loses that structure and forces the chunker to guess.
How big should each chunk be?
A common default is 800 tokens with 100-token overlap for prose. For code-heavy pages, smaller chunks (300-500 tokens) prevent semantically unrelated sections from being averaged together.
Should I store the raw HTML too?
Yes if you need to re-chunk later with different rules. Otherwise markdown plus the source URL is enough; you can re-fetch on demand.
