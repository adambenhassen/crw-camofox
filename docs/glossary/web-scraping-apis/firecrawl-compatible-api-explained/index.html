What is a Firecrawl-compatible API?

Short answer: A Firecrawl-compatible API mirrors Firecrawl's REST surface byte-for-byte: same endpoint paths, same JSON request body keys, and the same {"success": true, "data": {...}} response wrapper. The point is drop-in substitution — swap the base URL and your existing Firecrawl client keeps working with no code changes.

Which endpoints are covered

A complete Firecrawl-compatible surface implements five canonical endpoints:

  • POST /v1/scrape — single-URL extraction returning markdown, html, links, or structured JSON.
  • POST /v1/crawl — multi-page crawl, started as a job and polled via GET /v1/crawl/{id}.
  • POST /v1/extract — LLM-driven structured extraction (schema or prompt mode).
  • POST /v1/map — sitemap-style URL discovery for a domain.
  • POST /v1/search — web search with optional inline scrape of each result.

What "compatible" really means

True compatibility is not just URL parity. It also requires:

  • Request body parity: the same field names (url, formats, onlyMainContent, waitFor, limit, search, includeSubdomains) with the same defaults.
  • Response wrapper parity: a top-level success boolean plus a data object containing markdown, html, links, metadata, etc.
  • Auth header parity: the same Authorization: Bearer ... scheme on the hosted variant.
  • Error envelope parity: the same status codes and error JSON shape so retry logic does not need to branch on provider.

How fastCRW does it

fastCRW exposes the five endpoints above at https://api.fastcrw.com with identical request and response shapes. See the compatibility matrix and the full API reference. The Rust binary itself is AGPL-3.0 and runs at ~50 MB RAM idle, so the same surface is available self-hosted at http://localhost:3000.

Frequently asked

Can I point an existing Firecrawl SDK at fastCRW?
Yes. Most SDKs accept a base-URL override. Change it from https://api.firecrawl.dev to https://api.fastcrw.com, keep the same API key header, and existing code paths continue to work.
Are response fields exactly the same?
The shared fields (markdown, html, rawHtml, links, metadata, success, data) are byte-shape compatible. fastCRW adds a few extra fields like renderDecision and creditCost that Firecrawl clients ignore.
Does compatibility cover the crawl polling pattern?
Yes. POST /v1/crawl returns a job id and GET /v1/crawl/{id} returns status plus accumulated data entries, matching Firecrawl's polling contract.
